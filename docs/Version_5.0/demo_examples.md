# AI機能デモ例・RAGの仕組み解説

実際のコード（`backend/services/ai/`, `backend/repositories/vector_repository.py`）に基づく、
プロンプト例・出力例・RAGの仕組みの解説。

## 1. RAGとは何をしているのか(このアプリでの実装)

一般的なRAG(Retrieval-Augmented Generation)は「①検索 → ②プロンプトへの注入 → ③生成」の3段階。
本アプリの`POST /api/ai/search-assist`エンドポイントでの対応関係は以下の通り。

| 段階 | 処理 | 実装箇所 |
|---|---|---|
| ① Retrieval（検索） | ユーザーの質問文をそのままChromaDBに流し込む。ChromaDB内部で質問文を埋め込みベクトル化し、登録済みレシピの埋め込みとコサイン類似度で比較 | `vector_repository.search_similar_recipes()` |
| ② Augmented（コンテキスト注入） | 検索でヒットしたレシピの構造化テキスト(`build_recipe_document`で作成)を、プロンプト文字列にそのまま埋め込む | `ai_service.py` → `gemini_client.search_assist()` |
| ③ Generation（生成） | ②のプロンプトをLLMに渡し、「登録レシピを踏まえた」回答を生成させる | `GeminiClient._generate_text()` |

### レシピのベクトル化テキスト(検索対象)の構造

`build_recipe_document()`は、レシピをただの文字列連結ではなく**ラベル付き構造**でベクトル化用テキストにしている。以下は実際にアプリへ登録済みのレシピ「ボンゴレビアンコ」を例にした構造(材料整形部分は`format_ingredients_text()`として関数化・共通化されている)。

```
レシピ名: ボンゴレビアンコ
カテゴリ: イタリアン
調理時間: 13分
人数: 2人前
材料: スパゲッティ(1.6mm〜1.7mm推奨) 200g、アサリ(殻付き、砂出し済み) 300g、ニンニク 2片、赤唐辛子 1本、白ワイン(辛口) 80ml、オリーブオイル 大さじ3、パセリ(みじん切り) 大さじ2
手順: 1. 下準備をする 2. パスタを茹でる 3. ニンニクと唐辛子を炒める 4. アサリを加え蒸し煮にする 5. アサリを取り出す 6. ソースを乳化させる 7. パスタとアサリを戻し入れ、仕上げる
```

「材料:」「手順:」とラベルを付けることで、「材料から検索したい」質問と「調理法から検索したい」質問の両方に対応しやすくなる、というのが設計意図(コード内コメントより)。

## 2. 実際のプロンプト例

### 例A: `search-assist`(RAG横断検索)のプロンプト

> **注記(v5.0時点):** 本エンドポイントはバックエンドAPIとして実装済みで、以下は実際に動作確認したイメージだが、フロントエンドのUIとはまだ接続されていない(詳細は README「既知の課題と対応状況」を参照)。

ユーザーの質問: `「時短で作れる料理ある?」`

**① 検索結果(ChromaDBのヒット、イメージ)**

| recipe_id | title | score(コサイン距離) |
|---|---|---|
| 10 | ガスパチョ | 0.29 |
| 11 | ボンゴレビアンコ | 0.35 |

**② LLMに実際に渡されるプロンプト**(`gemini_client.py`の`search_assist`より)

```
以下の登録レシピを参照して回答してください。

【登録レシピ】
【レシピ1: ガスパチョ】
レシピ名: ガスパチョ
カテゴリ: 洋食
調理時間: 10分
人数: 2人前
材料: 完熟トマト 450g、きゅうり 0.5本、赤パプリカ 1/4個、玉ねぎ 1/8個、にんにく 1片、オリーブオイル 大さじ3
手順: 1. パンの下準備 2. 野菜のカット 3. 野菜をブレンダーに入れる（略）

【レシピ2: ボンゴレビアンコ】
レシピ名: ボンゴレビアンコ
カテゴリ: イタリアン
調理時間: 13分
人数: 2人前
材料: スパゲッティ(1.6mm〜1.7mm推奨) 200g、アサリ(殻付き、砂出し済み) 300g、ニンニク 2片（略）

質問: 時短で作れる料理ある?
```

**③ 出力(APIレスポンス、イメージ)**

```json
{
  "answer": "登録レシピの中では「ガスパチョ」(10分)と「ボンゴレビアンコ」(13分)が特に時短向きです。ガスパチョは加熱不要で野菜をブレンダーにかけるだけ、ボンゴレビアンコもパスタの茹で時間と並行して調理が進むため、平日でも作りやすい一皿です。",
  "is_mock": false,
  "references": [
    {"recipe_id": 10, "title": "ガスパチョ", "category": "洋食", "score": 0.29},
    {"recipe_id": 11, "title": "ボンゴレビアンコ", "category": "イタリアン", "score": 0.35}
  ]
}
```

`references`をフロントエンドで表示することで、「AIが何を根拠に回答したか」を利用者に見せられる（ハルシネーション対策の可視化）。ただしv5.0時点ではこの`references`を表示するUI自体が未実装のため、上記は仕様上想定される出力構造の説明である。

### 例A': `ai-assist`(個別レシピへのAI質問)の実際の動作結果

こちらは`search-assist`とは異なり、v5.0でフロントUIまで含めて動作確認済み。ローカルdev環境で「ガスパチョ」の詳細画面から質問した実際のやり取り。

ユーザーの質問: `「ケチャップマニスの代用品はある?」`

**LLMに渡されるコンテキスト**(`format_ingredients_text()`で整形):

```
レシピ名: ガスパチョ
材料: 完熟トマト 450g、きゅうり 0.5本、赤パプリカ 1/4個、玉ねぎ 1/8個、にんにく 1片、
      バゲット(または食パン) 30g、オリーブオイル 大さじ3、シェリービネガー 大さじ2、
      塩 大さじ1/2、冷水 適量
```

**実際に確認した出力:**

検証時点ではGoogle Gemini APIの呼び出しが認証エラー(`401 UNAUTHENTICATED`)で失敗し、既存のmockフォールバック機構により以下のようなモック回答が返る挙動を確認した。

```json
{
  "answer": "みりん → 砂糖小さじ1＋酒大さじ1で代用できます。",
  "is_mock": true
}
```

「代用」という語をキーワードマッチングした固定テンプレート応答であり、レシピ名や実際の材料(ガスパチョはみりんを使わない)とは無関係な回答になっている点が、モックフォールバックの限界として観察された。API未設定/障害時にアプリ全体を止めずに応答を返し続けられる一方、内容の妥当性そのものは保証しないという設計上のトレードオフが、実データでの検証によって具体的に確認できた形になる。

### 例B: `generate-recipe`(構造化出力)のプロンプト

`gemini_client.py`の`_GENERATE_PROMPT`は、GeminiにJSON以外を出力させない制約をプロンプトレベルとAPIパラメータレベルの二重で行っている。

```
「ボンゴレビアンコ」(2人前)のレシピをJSON形式のみで出力してください。
{"title":"ボンゴレビアンコ","category":"和食/洋食/中華/イタリアン/アジアン/副菜/その他のいずれか", ...}
```

APIパラメータ側でも`response_mime_type="application/json"`を指定しており(`GenerateContentConfig`)、プロンプトの指示とAPI側の構造化出力制約を組み合わせることで、パース失敗のリスクを下げている。

## 3. `is_mock`フラグの設計意図

すべてのAIエンドポイントのレスポンスに`is_mock: bool`が含まれる。これは`LLM_PROVIDER`が`mock`のとき、あるいはAPIキー未設定/API呼び出し失敗でモックにフォールバックしたときに`true`になる。

**なぜこの設計にしたか(README「テストにおける副作用の切り離し方針」より):**

- テスト実行時に本物のAPIを叩かないことを保証する仕組みが`llm_provider=mock`をデフォルト値にすることで実現している
- フロントエンド側で「これはデモ用のモック回答です」という表示を出し分けられる
- 障害時にAPIごと落ちるのではなく、モック回答で機能を継続提供できる(グレースフルデグラデーション)

**v5.0での実地検証:** `ai-assist`の動作確認中、実際にGemini API呼び出しが`401 UNAUTHENTICATED`で失敗し、`is_mock: true`のモック応答に自動フォールバックする様子をターミナルログ上で確認した。設計通りアプリ全体は落ちずに応答を返し続けたが、モック応答の内容自体はレシピ内容と無関係な固定テンプレートになりうる(上記「例A'」参照)ことも合わせて確認できた。

## 4. 簡易評価の観点

現状は自動化された精度評価(BLEU等）は導入していないが、`tests/test_ai.py`で以下を契約テストとして担保している。

- 各AIエンドポイントが`llm_provider=mock`で決定的なレスポンス形式を返すこと
- `search-assist`が`references`配列を含むレスポンス構造を維持すること

将来的な拡張案としては、検索結果の`score`(コサイン距離)分布を記録し、`score_threshold=1.2`の妥当性を実データで検証する、という方向性が考えられる(現時点では未実施)。
