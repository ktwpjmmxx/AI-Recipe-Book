# AI機能デモ例・RAGの仕組み解説

実際のコード（`backend/services/ai/`, `backend/repositories/vector_repository.py`）に基づく、
プロンプト例・出力例・RAGの仕組みの解説。

## 1. RAGとは何をしているのか(このアプリでの実装)

一般的なRAG(Retrieval-Augmented Generation)は「①検索 → ②プロンプトへの注入 → ③生成」の3段階。
本アプリの`POST /api/ai/search-assist`エンドポイントでの対応関係は以下の通り。

| 段階 | 処理 | 実装箇所 |
|---|---|---|
| ① Retrieval（検索） | ユーザーの質問文をそのままChromaDBにcrossフィード。ChromaDB内部で質問文を埋め込みベクトル化し、登録済みレシピの埋め込みとコサイン類似度で比較 | `vector_repository.search_similar_recipes()` |
| ② Augmented（コンテキスト注入） | 検索でヒットしたレシピの構造化テキスト(`build_recipe_document`で作成)を、プロンプト文字列にそのまま埋め込む | `ai_service.py` → `gemini_client.search_assist()` |
| ③ Generation（生成） | ②のプロンプトをLLMに渡し、「登録レシピを踏まえた」回答を生成させる | `GeminiClient._generate_text()` |

### レシピのベクトル化テキスト(検索対象)の構造

`build_recipe_document()`は、レシピをただの文字列連結ではなく**ラベル付き構造**でベクトル化用テキストにしている。

```
レシピ名: 豚キムチ炒め
カテゴリ: 和食
調理時間: 15分
人数: 2人前
材料: 豚バラ肉 200g、キムチ 100g、ごま油 大さじ1
手順: 1. 豚肉を炒める 2. キムチを加えて炒め合わせる 3. ごま油で香り付けする
```

「材料:」「手順:」とラベルを付けることで、「材料から検索したい」質問と「調理法から検索したい」質問の両方に対応しやすくなる、というのが設計意図(コード内コメントより)。

## 2. 実際のプロンプト例

### 例A: `search-assist`(RAG横断検索)のプロンプト

ユーザーの質問: `「キムチを使った時短レシピある?」`

**① 検索結果(ChromaDBのヒット、イメージ)**

| recipe_id | title | score(コサイン距離) |
|---|---|---|
| 12 | 豚キムチ炒め | 0.31 |
| 27 | キムチ豆腐チゲ | 0.42 |

**② LLMに実際に渡されるプロンプト**(`gemini_client.py`の`search_assist`より)

```
以下の登録レシピを参照して回答してください。

【登録レシピ】
【レシピ1: 豚キムチ炒め】
レシピ名: 豚キムチ炒め
カテゴリ: 和食
調理時間: 15分
人数: 2人前
材料: 豚バラ肉 200g、キムチ 100g、ごま油 大さじ1
手順: 1. 豚肉を炒める 2. キムチを加えて炒め合わせる 3. ごま油で香り付けする

【レシピ2: キムチ豆腐チゲ】
（略）

質問: キムチを使った時短レシピある?
```

**③ 出力(APIレスポンス)**

```json
{
  "answer": "登録レシピの中では「豚キムチ炒め」が15分で作れて時短向きです。豚バラ肉とキムチをごま油で炒めるだけのシンプルな工程なので、平日の夕食にも向いています。",
  "is_mock": false,
  "references": [
    {"recipe_id": 12, "title": "豚キムチ炒め", "category": "和食", "score": 0.31},
    {"recipe_id": 27, "title": "キムチ豆腐チゲ", "category": "和食", "score": 0.42}
  ]
}
```

`references`をフロントエンドで表示することで、「AIが何を根拠に回答したか」を利用者に見せられる（ハルシネーション対策の可視化）。

### 例B: `generate-recipe`(構造化出力)のプロンプト

`gemini_client.py`の`_GENERATE_PROMPT`は、GeminiにJSON以外を出力させない制約をプロンプトレベルとAPIパラメータレベルの二重で行っている。

```
「豚キムチ炒め」(2人前)のレシピをJSON形式のみで出力してください。
{"title":"豚キムチ炒め","category":"和食/洋食/中華/イタリアン/アジアン/副菜/その他のいずれか", ...}
```

APIパラメータ側でも`response_mime_type="application/json"`を指定しており(`GenerateContentConfig`)、プロンプトの指示とAPI側の構造化出力制約を組み合わせることで、パース失敗のリスクを下げている。

## 3. `is_mock`フラグの設計意図

すべてのAIエンドポイントのレスポンスに`is_mock: bool`が含まれる。これは`LLM_PROVIDER`が`mock`のとき、あるいはAPIキー未設定/API呼び出し失敗でモックにフォールバックしたときに`true`になる。

**なぜこの設計にしたか(README「テストにおける副作用の切り離し方針」より):**

- テスト実行時に本物のAPIを叩かないことを保証する仕組みが`llm_provider=mock`をデフォルト値にすることで実現している
- フロントエンド側で「これはデモ用のモック回答です」という表示を出し分けられる
- 障害時にAPIごと落ちるのではなく、モック回答で機能を継続提供できる(グレースフルデグラデーション)

## 4. 簡易評価の観点

現状は自動化された精度評価(BLEU等）は導入していないが、`tests/test_ai.py`で以下を契約テストとして担保している。

- 各AIエンドポイントが`llm_provider=mock`で決定的なレスポンス形式を返すこと
- `search-assist`が`references`配列を含むレスポンス構造を維持すること

将来的な拡張案としては、検索結果の`score`(コサイン距離)分布を記録し、`score_threshold=1.2`の妥当性を実データで検証する、という方向性が考えられる(現時点では未実施)。
